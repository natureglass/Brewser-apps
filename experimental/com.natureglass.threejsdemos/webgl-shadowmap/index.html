Three.js r162 — Shadow Map

Port of webgl_shadowmap.html: a ground plane with 6 primitive caster shapes (boxes, spheres, torus) under AmbientLight + DirectionalLight with castShadow. renderer.shadowMap.enabled = true and renderer.shadowMap.type = PCFShadowMap activate Three.js's 17-tap PCF shadow path. The auto-generated MeshPhongMaterial shadow-receive shader and the MeshDepthMaterial shadow-cast shader BOTH route to the bridge's raw-shader passthrough path via the new #define USE_SHADOWMAP / #define DEPTH_PACKING auto-promote scanners (milestone #20 nxjs-source extension). Also forced a critical fix in gl.uniform1i — it was silently rejecting bool uniforms (like Three.js's receiveShadow) + sampler bindings to non-zero texture units; the receive shader's shadow gate stayed false by default, hiding all shadows. The bridge auto-promotes any Three.js shader containing #define USE_SHADOWMAP (the shadow-receive forward shaders) or #define DEPTH_PACKING (the MeshDepthMaterial shadow-cast pass) to native-GLES dispatch via the same raw-shader passthrough mechanism that powers ShaderMaterial demos. Both halves of the shadow pipeline thus run as Three.js generates them — PCF kernel, depth packing, frustum check — without the bridge swapping in its own program at draw time. The WebGLShadowMap path needs the FBO surface from milestone #19 and the raw_passthrough path from milestone #14; it does NOT need WEBGL_depth_texture (the shadow map is RGBA UByte with depth RGBA-packed via packDepthToRGBA).
