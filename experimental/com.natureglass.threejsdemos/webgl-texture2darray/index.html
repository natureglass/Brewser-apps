Three.js r184 — 2D Texture Array

1:1 port of webgl_texture2darray.html from three.js latest (r184). Uploads a 256×256×109 8-bit RED volume (CT head scan by Divine Augustine, CPOL) as a single DataArrayTexture bound to TEXTURE_2D_ARRAY; a custom GLSL3 ShaderMaterial samples the array with texture(diffuse, vec3(uv, depth)) where depth animates 0→109 to sweep through the slices. Validates the WebGL 2 sampler2DArray + texImage3D surface on Citron Mesa Nouveau — different fragment-sampling code path from #34 (texture3d) which is parked on a vertex-texelFetch bug, so a clean run here proves the array-texture path is not co-affected.


No gamepad input — the demo just animates depth through the volume.

Pre-authorized deviations from upstream: (1) Three.js r184 loaded via the __THREE_R184_STAGED__ IIFE + the three nx.js shims at libs/r184-nxjs-bridge.js. (2) Canvas reused from <canvas id="t2da-canvas">; no document.createElement('div') / resize listener / setPixelRatio. (3) WebGL 2 context with the constructor.name = 'WebGL2RenderingContext' Proxy. (4) fflate.unzipSync dropped: the volume was pre-decompressed offline (Python zipfile) to a raw 7,143,424-byte .bin sidecar; loaded via Switch.readFile instead of FileLoader (fetch is blocked under r184 IIFE per [[r184-fetch-hang]]). (5) console.warn/log/error/info silenced. (6) renderer.resetState() before every render.
