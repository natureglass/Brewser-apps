Three.js r162 — Instancing Dynamic

Port of webgl_instancing_dynamic.html: 1000 instances of the Suzanne head mesh, each animated with a per-frame matrix update via InstancedMesh.setMatrixAt. Material is MeshNormalMaterial (normal-as-color output). The whole scene draws in a single drawElementsInstancedANGLE call — this milestone introduces native instancing through the bridge's raw-shader passthrough path.


Conformance: extensions/angle-instanced-arrays — ported upstream Khronos test for the ANGLE_instanced_arrays extension. Verifies the extension object shape (3 methods + the VERTEX_ATTRIB_ARRAY_DIVISOR_ANGLE constant), range checks on vertexAttribDivisorANGLE, error conditions on both draw entry points (count/primcount < 0, disallowed primitive modes, count==0/primcount==0 no-ops, UNSIGNED_BYTE indices), and end-to-end rendered output for both drawArraysInstancedANGLE and drawElementsInstancedANGLE — 4 quadrants colored red/green/blue/yellow via per-instance offset and color attributes (divisor=1). The output draws exercise the auto-promote-to-passthrough gate ([[bridge-instancing-support]]): no #pragma raw_passthrough needed in the shader sources, the divisor>0 setup is what routes through native dispatch.

Look for P+countXinst tags in the bridge debug line — those are the instanced passthrough dispatches. A single P+2901x1000 per frame indicates the whole 1000-Suzanne scene is drawing in one native glDrawElementsInstanced. At amount=10 (upstream default), expect ~1000 instances. Three.js issues one such draw per render. Performance depends entirely on the GPU's ability to chew through 967K triangles per frame; the CPU side only does per-instance matrix math and an instanceMatrix bufferSubData upload.
