Three.js r162 — Post-Processing

Port of webgl_postprocessing.html: 100 flat-shaded white spheres tumbling on a black-fog backdrop, run through an EffectComposer chain of RenderPass → DotScreenShader → RGBShiftShader → OutputPass. First milestone to exercise nx.js's new WebGLRenderTarget / FBO support — each intermediate pass renders into a real native GLES framebuffer and the next pass samples its color texture. The post-process shaders run end-to-end on native GLES via the #pragma raw_passthrough opt-in (bridge-raw-shader-passthrough). Two pre-authorized deviations: (1) the pragma is injected into the four stock post-process shaders at script-load time by the libs/effect-composer.js bundle; (2) intermediate RenderTargets default to UByte storage instead of upstream's HalfFloatType (no half-float color-buffer exposure yet on the bridge).


Conformance: renderbuffers/framebuffer-object-attachment — port of the Khronos FBO conformance test that drives every combination of framebufferTexture2D + framebufferRenderbuffer + checkFramebufferStatus and validates error returns for invalid attachment targets / formats / dimensions.

No interactive controls — pure animation. The EffectComposer chain ping-pongs between two WebGLRenderTargets (rt1 ↔ rt2), each a 640×360 RGBA8 texture + DEPTH_COMPONENT16 renderbuffer. RenderPass writes the scene into readBuffer, the two ShaderPasses read from there and write to writeBuffer (swapping each time), and the final OutputPass with renderToScreen = true binds null and composites to the bridge FBO — which then flows through the standard flush_bridge_present readback path to the screen canvas. Watch the status canvas's "FBO status" row for the COMPLETE verification, and "bridge dbg" for the per-draw dispatch tag stream (you'll see T+N for the scene's textured triangles routing into the user FBO, then P+N tags for the passthrough shader-pass draws).
